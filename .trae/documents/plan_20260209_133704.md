# macOS 磁盘扫描工具修复与优化计划

经过全面的代码审查，我发现了几个关键问题，包括严重的安全限制 Bug、性能瓶颈以及架构缺陷。以下是详细的修复计划。

## 1. 关键问题分析

### 1.1 安全性 Bug：用户无法删除文件
- **位置**: `src-tauri/src/commands.rs` 中的 `is_sensitive_path` 函数。
- **现象**: `/Users` 被列为敏感前缀，导致 `/Users` 下的所有文件（包括用户自己的文档）都被禁止删除。
- **修复**: 修改路径检查逻辑，仅保护系统关键目录（如 `/System`, `/bin`）和用户根目录本身（`/Users/username`），但允许删除用户目录下的子文件。

### 1.2 性能与架构问题
- **前端事件轰炸**: `scan_directory_with_progress` 每发现一个文件就向前端发送一个事件。对于包含数十万文件的磁盘，这将导致前端 UI 线程阻塞甚至崩溃。
- **后端锁竞争**: 并行扫描 (`scan_directory_parallel`) 使用全局锁 (`Mutex<HashMap>`) 来更新父目录大小，导致多线程优势被锁竞争抵消。
- **无法取消**: 并行扫描实现中没有检查取消标志，一旦开始无法停止。
- **算法低效**: 实时计算目录大小（向上遍历父目录）在深层嵌套结构中效率低下。

## 2. 修复方案

### 2.1 后端重构 (`src-tauri/src/scanner.rs`)
我们将重写扫描核心，采用"先收集，后聚合"的策略，彻底消除锁竞争。

1.  **统一扫描引擎**: 合并串行和并行扫描逻辑，统一使用基于 `Rayon` 的并行扫描器。
2.  **两阶段算法**:
    -   **阶段一 (并行 I/O)**: 并行遍历文件系统，收集所有文件信息 (`FileInfo`) 到线程局部向量，同时更新原子计数器用于进度报告。不在此阶段计算目录大小。
    -   **阶段二 (内存聚合)**: 扫描结束后，在内存中遍历收集到的文件列表，计算所有目录的大小。这是一个纯内存操作，速度极快，无需复杂的锁。
3.  **支持取消**: 在并行迭代器中嵌入取消标志检查。
4.  **优化进度报告**: 仅通过原子计数器定期发送进度概览（已扫描数量、总大小），不再发送单个文件的详细信息。

### 2.2 命令层修复 (`src-tauri/src/commands.rs`)
1.  **修正 `delete_path`**: 实现更细粒度的权限检查。
2.  **更新 `scan_directory_with_progress`**: 对接新的扫描引擎，调整事件发送策略（只发 `Progress` 和 `Completed`）。

### 2.3 前端优化 (`src/App.tsx`)
1.  **移除实时列表更新**: 停止监听 `fileFound` 和 `directoryFound` 事件（后端也不再发送）。
2.  **优化状态管理**: 仅在收到 `completed` 事件后一次性更新文件列表。
3.  **保留进度展示**: 继续处理 `progress` 事件以显示扫描进度条。

## 3. 实施步骤

1.  **重构 Scanner**: 修改 `scanner.rs`，实现新的并行扫描算法和目录大小聚合逻辑。
2.  **修复 Commands**: 修改 `commands.rs`，修复删除权限检查，并适配新的扫描接口。
3.  **清理前端**: 修改 `App.tsx`，移除冗余的事件监听代码。
4.  **验证**: 验证扫描速度、取消功能、删除功能以及大目录处理能力。
